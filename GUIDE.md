# 開発提案スキルセット 使い方ガイド

> Web・システム開発の受注から提案まで、Claude と一緒に進めるためのスキル群です。

---

## スキル一覧

| スキル名 | 役割 |
|---------|------|
| `market-researcher` | クライアント企業の市場調査・企業分析 |
| `engineer` | 技術選定・アーキテクチャ・DB/API設計 |
| `design` | UI/UX方針・カラー・画面構成の提案 |
| `estimator` | 工数・費用・期間の見積もり |
| `proposal-creator` | 提案書（PowerPoint形式）の作成 |

---

## 基本的な使い方

### 1. まず市場調査からスタート

クライアントの企業名を伝えるだけで調査が始まります。

```
「株式会社〇〇の市場調査をして」
「□□という会社について調べて」
「△△社、受注前にリサーチしておきたい」
```

調査結果として以下が出力されます：

- 事業概要・企業規模
- ブランド・デザイントーン
- 推定される課題・ペインポイント
- 開発提案へのヒント
- **後続スキルへの引き継ぎデータ（CLIENT_CONTEXT）** ← これが連携の核

---

### 2. 発注が来たら、そのまま続けて話しかける

市場調査が終わった会話の続きで依頼を伝えます。前の調査内容を自動で引き継ぎます。

```
「〇〇社から予約管理システムを作ってほしいと言われた。技術選定して」
「ECサイトのリニューアルを依頼された。見積もりを出してほしい」
「提案書を作りたい」
```

---

### 3. スキルは好きな順番・組み合わせで使える

全部使う必要はありません。必要なものだけ使えばOKです。

**よくある流れ（フル活用）**:
```
市場調査 → 技術設計 → デザイン方針 → 見積もり → 提案書作成
```

**部分的な使い方の例**:
```
「技術選定だけしてほしい」→ engineer スキルだけ使う
「デザインの方向性だけ決めたい」→ design スキルだけ使う
「見積もりだけ出してほしい」→ estimator スキルだけ使う
「提案書をすぐ作りたい」→ proposal-creator スキルだけ使う
```

## 各スキルの詳細

### market-researcher（市場調査）

**こんなときに使う**：
- 受注前にクライアントのことを調べておきたい
- ブランドトーンや業界課題を把握して提案精度を上げたい

**話しかけ方の例**：
```
「〇〇社を調べて」
「このクライアントについてリサーチして」
「受注前に調べておきたい」
```

**出力されるもの**：
- 事業概要・企業規模
- ブランド・デザイントーン
- 推定される課題・ペインポイント
- 開発提案へのヒント
- 後続スキルへの引き継ぎデータ（CLIENT_CONTEXT）

---

### engineer（技術設計）

**こんなときに使う**：
- どの技術スタックを使うか決めたい
- システム構成・DB・APIを設計したい

**話しかけ方の例**：
```
「技術選定して」
「アーキテクチャを設計して」
「DB設計して」
「システム構成を考えて」
```

**出力されるもの**：
- 推奨技術スタック（フロントエンド・バックエンド・DB・インフラ）
- システムアーキテクチャ図
- 主要テーブル定義
- 主要APIエンドポイント
- セキュリティ・非機能要件

---

### design（UI/UXデザイン）

**こんなときに使う**：
- デザインの方向性を決めたい
- 画面構成やカラーパレットを提案してほしい

**話しかけ方の例**：
```
「デザインの方向性を決めて」
「UI設計して」
「画面構成を考えて」
「カラーパレットを提案して」
```

**出力されるもの**：
- デザインコンセプト（一言の方向性）
- カラーパレット（用途別）
- タイポグラフィ・コンポーネント方針
- 主要画面のワイヤーフレーム（テキスト形式）
- レスポンシブ対応方針

---

### estimator（見積もり）

**こんなときに使う**：
- クライアントに費用感を提示したい
- 工数・期間の目安を出したい
- 提案書に見積もりを入れたい

**話しかけ方の例**：
```
「見積もりを出して」
「工数と費用の概算がほしい」
「どのくらいの期間とお金がかかる？」
```

**出力されるもの**：
- 工程別の工数表（人日）
- 費用見積もり（開発費・インフラ費・ランニングコスト）
- 機能別の工数内訳
- スケジュール概算
- 前提・注意事項
- 全スキルの成果物チェックリスト

---

### proposal-creator（提案書作成）

**こんなときに使う**：
- クライアントへの提案書（PowerPoint）を作りたい
- 自社フォーマットに合わせた資料を作りたい

**話しかけ方の例**：
```
「提案書を作って」
「このテンプレートで提案書を作って」
「顧客向けの提案資料を仕上げて」
```

**出力されるもの**：
- PowerPoint形式の提案書ファイル（.pptx）
- 表紙・課題認識・提案内容・費用・スケジュール等のスライド構成
- 自社フォーマットがある場合はそれに忠実な再現

> 自社のフォーマットファイル（.pptx / .docx / .pdf）を添付して渡すと、そのデザインを踏襲して作成します。

## よくある質問

**Q. 必ず市場調査から始めないといけない？**
A. いいえ。市場調査なしでも各スキルは動きます。ただし、市場調査をしておくと後続スキルの提案精度が上がります（ブランドトーン・課題を引き継ぐため）。

**Q. 別の会話（セッション）に引き継げる？**
A. 現在のスキルは同じ会話の中での引き継ぎを前提にしています。別の会話で使うときは「さっき市場調査した〇〇社の情報として…」と要点をコピペして伝えると、同様に活用できます。

**Q. 日本語以外の企業も調査できる？**
A. できます。market-researcher スキルは日本語・英語の両方で検索します。「海外の会社です」と伝えると精度が上がります。

**Q. スキルのカスタマイズはできる？**
A. 各スキルフォルダ内の `SKILL.md` はテキストファイルなので、直接編集して調整できます。例えば「estimator」の単価前提を自社の相場に合わせて書き換えることが可能です。

**Q. スキルをインストールしたら、毎回スキル名を指定して呼び出す必要がある？**
A. いいえ、不要です。スキルをインストールしておくと、Claude が会話の内容を読んで**自動で適切なスキルを起動**します。「〇〇社を調べて」と言えば market-researcher が、「見積もりして」と言えば estimator が自動で動きます。スキル名を意識して呼ぶ必要はありません。
